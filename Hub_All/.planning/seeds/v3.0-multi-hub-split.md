## 1. Bối cảnh & vấn đề

Hub_All hiện tại = **1 FastAPI process, 1 DB `medinet_central`, multi-tenancy LOGICAL theo `hub_id`**. 3 hub (y_te, dược, HCNS) cùng table, phân lập bởi `WHERE hub_id = ?` ở repository layer (Phase 5 — E4 critical test PASS).

User decision 2026-05-21: tách các hub con ra **process + DB riêng**, hub tổng đóng vai trò aggregator nhận chunks/vector từ hub con để search cross-hub tập trung.

**Lý do nghiệp vụ (suy ra từ requirement, chốt chính thức ở /gsd-spec-phase):**
- Mỗi hub con sẽ có chủ sở hữu (Hub y_te = phòng y tế Medinet, Hub dược = phòng dược, ...), cần isolation vật lý mạnh hơn `WHERE hub_id`.
- URL `wiki.domain.com/<ten_hub>` cho phép từng hub có brand/permission riêng nhưng vẫn dưới domain Medinet.
- Hub con KHÔNG cần (và KHÔNG được phép) đụng vào system settings (rag-config provider, api_keys global, hub registry) — chỉ admin Medinet trên hub tổng được làm.

## 2. Architectural decisions (đã chốt 2026-05-21)

> Các quyết định dưới đây đã LOCKED qua AskUserQuestion 2026-05-21. Đảo chiều cần explicit re-discuss.

### D-V3-01: DB topology = "Postgres database riêng cùng instance"
- `medinet_central` (hub tổng) + `medinet_hub_yte` + `medinet_hub_duoc` + `medinet_hub_hcns` + ... trên **1 instance Postgres**.
- Backup chung, network đơn giản, cocoindex flow ngắn (mỗi hub con có cocoindex flow riêng đẩy vào DB local + hook đẩy chunks lên `medinet_central`).
- Bỏ phương án "schema riêng per hub" (đụng R5/P7 — cocoindex `db_schema_name` cố định) và "instance riêng per hub" (ops × N).

### D-V3-02: Dataflow hub con → hub tổng = "Chunks + vector"
- Sau khi cocoindex flow của hub con ingest xong (extract → chunk → embed local DB), **push chunks + vector lên `medinet_central.chunks`** (denormalized read replica).
- Hub tổng KHÔNG re-embed — nhận sẵn vector 1536-dim từ hub con (cùng provider/model do hub tổng quyết định qua rag-config).
- Cost: storage 2× (vector lưu 2 chỗ); benefit: cross-hub search 1 DB nhanh, không federated HTTP fan-out.
- Sync direction: 1 chiều **hub con → tổng**. Hub tổng KHÔNG ghi ngược về hub con.
- Cơ chế cụ thể (Postgres logical replication / cocoindex target thứ 2 / outbox + worker) chốt ở /gsd-discuss-phase.

### D-V3-03: Scoping = "Milestone v3.0 mới"
- Đây là milestone-level work, KHÔNG nhét vào M2 dưới dạng 1 phase đơn lẻ.
- Triggering: `/gsd-new-milestone` sau khi M2 done (Phase 4/8.3/9/10).
- Tên đề xuất: `v3.0 — Multi-Hub Split` (sửa lại tên "Multi-subdomain SPA" trong PROJECT.md vì user chọn **subpath** thay vì **subdomain** — `wiki.domain.com/<hub>` thay vì `<hub>.wiki.domain.com`).

### D-V3-04: M2 closeout = bắt buộc trước v3.0

## 3. Gray areas CÒN MỞ (chốt ở /gsd-spec-phase v3.0 sau)

> Đã trả lời 4 câu lớn ở section 2. Còn 4 câu chưa rõ — cần chốt khi mở milestone:

### GA-V3-A: Auth & SSO
- User login 1 lần ở hub tổng → JWT chung valid trên hub con? Hay mỗi hub con có user table riêng?
- Nếu JWT chung: hub con verify JWT bằng cách nào (shared keypair? JWKS endpoint từ hub tổng?). Refresh token race (P16) ở hub nào?
- Hub isolation E4 hiện = `WHERE hub_id` ở repo layer. Sau split, hub con CHỈ có data của chính nó → E4 đơn giản hơn. Cross-hub search ở hub tổng cần `hub_ids: list[str]` trong JWT.

### GA-V3-B: System settings sync
- `rag_config` (LLM/embedding provider), `api_keys` (global), `hub_registry` sống ở hub tổng. Hub con đọc thế nào?
  - Option a: HTTP fetch on-demand (`GET https://hub-tong/api/rag-config` mỗi lần ingest)
  - Option b: Push sync (hub tổng push config change xuống hub con qua webhook/Redis pub/sub)
  - Option c: Env var local (operator deploy lại hub con khi đổi provider)
- Trade-off: latency vs simplicity vs consistency.

### GA-V3-C: Reverse proxy + frontend prefix
- Caddy/nginx route `wiki.domain.com/<hub>/*` → `http://hub-<hub>:8180/*` (strip prefix `<hub>`).
- Frontend SPA: 1 build dùng chung, detect prefix từ `window.location.pathname.split('/')[1]` để gắn vào API URL? Hay build per-hub với env `VITE_HUB_NAME=yte`?
- `frontend/src/services/api.ts` line 3 hardcode `:8180` — sẽ phá khi split. Frontend cần đổi base URL detect prefix.
- D6 ("KHÔNG sửa frontend") đã được nâng lên thành "fix CSS được phép sửa React" (memory feedback). Split URL chắc chắn cần sửa frontend → cần explicit re-confirm D6 expiry ở v3.0.

### GA-V3-D: Migration data từ `medinet_central` cũ
- DB tổng đang có `hub_yte/duoc/hcns` chung. Split = chia data:
  - Option a: `pg_dump` per `WHERE hub_id` → restore vào DB hub con; truncate `hub_id` rows khỏi tổng (giữ skeleton aggregate).
  - Option b: Snapshot + replay cocoindex flow trên hub con (rebuild from `file_store`).
- MCP Service hiện trỏ về 1 API → sau split phải re-point (gọi hub tổng cho aggregate, hay fan-out N hub con?).

## 5. Risks & EXIT criteria (preview)

- **R-V3-1 (HIGH)**: Sync chunks + vector 2 lần (hub con + tổng) gây drift → query mismatch. Mitigation: outbox pattern + checksum verify periodic.
- **R-V3-2 (HIGH)**: Frontend sửa lớn → vi phạm D6 hiện tại. Mitigation: D6 expire chính thức ở v3.0-05.
- **R-V3-3 (MEDIUM)**: Per-hub Alembic drift (mỗi DB tự upgrade) → schema mismatch giữa các hub. Mitigation: CI lint + Alembic check tự động mọi DB.
- **R-V3-4 (MEDIUM)**: Migration data từ DB cũ — downtime cần thiết. Mitigation: blue/green per-hub.

**EXIT v3.0 (đề xuất, chốt ở /gsd-roadmapper):**
- E-V3-1: 3 hub con + tổng healthy trên Docker Compose; URL `wiki.domain.com/yte` PASS golden path (login → upload → search local + cross-hub)
- E-V3-2: Cross-hub search latency p95 < 1.5s (không regress so với Phase 6)
- E-V3-3: Hub con KHÔNG truy cập được data hub khác (DB-level isolation — E4 reinforced)
- E-V3-4: System settings change ở tổng → propagate xuống hub con trong < 30s
