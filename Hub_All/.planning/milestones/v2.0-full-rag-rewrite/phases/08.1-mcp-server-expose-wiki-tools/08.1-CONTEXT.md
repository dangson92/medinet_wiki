<decisions>
## Implementation Decisions

### MCP Architecture (Transport & triển khai)
- **D-01:** Transport = **Streamable HTTP** (transport MCP hiện đại, thay thế SSE). KHÔNG dùng stdio — wiki là server hosted multi-Hub, client kết nối từ xa vào, không spawn local process.
- **D-02:** MCP server **mount vào CHÍNH FastAPI app hiện hữu** như sub-application (đề xuất path `/mcp`), KHÔNG tiến trình riêng. Tái dùng uvicorn process port 8180, lifespan, `app.state.db_pool`, `app.state.redis`. Lý do: tiến trình riêng phải nhân đôi wiring DB pool / cocoindex / config — nhất quán ARCHITECTURE Pattern 1 (in-process, single FastAPI process owns mọi thứ).
- **D-03:** SDK = **MCP Python SDK chính thức** (`mcp`, gồm FastMCP). Thêm dependency `mcp` vào `api/pyproject.toml` — researcher pin version cụ thể tương thích `fastapi==0.136.1` + Python 3.12 (pin exact, không bump giữa phase — chống P19).
- **D-04:** MCP tool gọi **TRỰC TIẾP service layer** (`AskService` / `SearchService`) trong cùng process — KHÔNG self-call HTTP `/api/*`. Service layer chạy trên asyncpg pool thuần (không cần FastAPI Depends / AsyncSession cho search/ask).

### Authentication (Xác thực MCP client)
- **D-05:** Xác thực MCP client = **API key qua HTTP header `X-API-Key`**. Tái dùng `ApiKeyService.verify_key` (Phase 5 AUX-02) → resolve `api_keys.created_by` → load `User`. KHÔNG dùng OAuth 2.1 — quá nặng cho công cụ nội bộ + áp lực R3 anti-pivot fatigue.
- **D-06:** Streamable HTTP transport mang HTTP header → MCP client config khai báo custom header `X-API-Key: <key>`. Key sai / thiếu / đã thu hồi → MCP error (401 ở transport level), KHÔNG để tool chạy.
- **(research):** Researcher xác nhận Claude Desktop / ChatGPT connectors / Cursor có hỗ trợ custom header trên Streamable HTTP connection. Nếu một client KHÔNG hỗ trợ → ghi nhận hạn chế trong RESEARCH.md, KHÔNG mở rộng OAuth trong 8.1.

### Tool Surface (Bộ công cụ expose)
- **D-07:** Expose **3 tool read-only**:
  - `search_wiki` — vector search. Params: `query` (bắt buộc), `hub_id` (tuỳ chọn), `top_k` (tuỳ chọn, default 10). Trả chunks: content, score, document name, hub name.
  - `ask_wiki` — RAG answer + citation. Params: `query` (bắt buộc), `hub_id` (tuỳ chọn). Trả `answer` + mảng `citations`.
  - `list_hubs` — trả danh sách hub mà user của API key được phép (id + name). Lý do: client cần biết `hub_id` hợp lệ để dùng param `hub_id` — đây là helper enable search/ask, KHÔNG phải năng lực mới.
- **D-08:** `hub_id` là tham số **TUỲ CHỌN** trên `search_wiki` / `ask_wiki`: có giá trị → search/ask trong hub đó; bỏ trống → cross-hub trên MỌI hub user được phép. Gộp single/cross-hub vào 1 tool mỗi loại — đơn giản cho LLM client. Nội bộ route sang `search` vs `search_cross_hub` / `ask` vs `ask_cross_hub`.
- **D-09:** KHÔNG expose MCP **resources & prompts** trong 8.1 — chỉ tools. Roadmap nói "tool". Resource browsing (duyệt hub/document) là phase tương lai.
- **D-10:** KHÔNG expose `find_similar` — cần `chunk_id` mà external client không có sẵn.
- **D-11:** Citation: `ask_wiki` trả **structured tool output** gồm (a) `answer` text giữ NGUYÊN marker `[N]` (đọc được trực tiếp), (b) mảng `citations` structured (`chunk_id`, `document_name`, `hub_name`, `snippet`, `score`) để LLM client attribute nguồn.

### Hub Isolation & Usage Logging
- **D-12:** Hub isolation derive từ API key: `api_keys.created_by` → `User` → `user_hubs` assignments. MCP server tự dựng object tương đương `UserWithHubs` (load user + hub_ids thủ công — không có FastAPI Depends) rồi truyền xuống `AskService` / `SearchService`. Enforcement hub isolation ĐÃ CÓ sẵn ở service layer (intersect hub_ids TRƯỚC SQL) — KHÔNG viết logic isolation mới.
- **D-13:** `hub_id` là **filter, KHÔNG phải escalation**: client truyền `hub_id` ngoài phạm vi user được phép (và không phải admin) → service layer reject (HubIsolationError) → MCP error tường minh. Bỏ trống = mọi hub được phép. admin role → cross-hub mọi hub active (giữ behavior service layer).
- **D-14:** `ask_wiki` **LOG `usage_events`** (nhất quán với `/api/ask` — ASK-05). `user_id` = user của API key; `hub_id` có thể null khi cross-hub. Ghi non-blocking.

### Claude's Discretion
- Mount path cụ thể (`/mcp` đề xuất), nội dung tool description text LLM-facing, format MCP JSON-RPC error response, có ghi `audit_logs` cho mỗi tool call hay không, rate limiting cho MCP tool (cân nhắc tái dùng ngưỡng search 100/min). Researcher/planner tự quyết trong khuôn khổ các quyết định trên.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `AskService` / `SearchService`: chạy trên asyncpg pool thuần (không cần AsyncSession) — MCP tool gọi trực tiếp được, không cần HTTP self-call.
- `ApiKeyService.verify_key`: verify X-API-Key sẵn sàng — auth MCP client tái dùng nguyên.
- `UserWithHubs` + mẫu query `user_hubs` trong `get_current_user_with_hubs`: MCP server clone logic này (thủ công, không Depends) để dựng object truyền xuống service.
- `log_usage_event`: token usage logging sẵn — `ask_wiki` tái dùng.
- `HubIsolationError`: hub isolation enforce ở service/repo layer — MCP không cần logic mới.

### Established Patterns
- Single FastAPI process owns mọi thứ (in-process cocoindex, ARCHITECTURE Pattern 1) → MCP server mount cùng process, cùng lifespan, cùng `app.state`.
- Router mount qua `app.include_router(...)` trong `create_app()` — MCP sub-app mount cạnh đó (Streamable HTTP MCP app là ASGI app, mount qua `app.mount("/mcp", ...)`).
- App port 8180 (frontend hardcode); MCP endpoint nằm dưới cùng port, path `/mcp`.

### Integration Points
- `create_app()` trong `main.py` — thêm bước mount MCP ASGI sub-app.
- `lifespan` — MCP server cần `app.state.db_pool` (đã init ở lifespan step 1). Không cần thêm lifespan resource mới.
- `pyproject.toml` dependencies — thêm `mcp` (pin version).

</code_context>
