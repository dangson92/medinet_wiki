## Critical Issues

### CR-01: `asyncio.create_task` không được track — mất bản ghi usage khi shutdown nhanh (D-14)

**File:** `Hub_All/api/app/mcp/server.py:241`

**Vấn đề:** `asyncio.create_task(log_usage_event(...))` bị suppress bằng `# noqa: RUF006` (RUF006 = "asyncio task is not stored"). Điều này có nghĩa là nếu server đang shutdown (hoặc nếu event loop cancel pending tasks), task ghi `usage_events` bị silently drop mà KHÔNG có cơ chế flush/cancel-aware như `BackgroundTasks` của FastAPI. Khác với pattern HTTP router (dùng `background_tasks.add_task` — FastAPI đảm bảo task chạy trước khi response close), trong MCP tool không có lifecycle tương đương.

Ngoài ra, nếu `log_usage_event` raise exception trong task background, exception đó sẽ bị bỏ qua hoàn toàn (unhandled task exception) — không có error log, không có retry.

Đây là vấn đề **data integrity** (usage billing/audit) và **silent exception swallowing** — xếp Critical vì thông tin billing có thể mất mà không có cảnh báo.

```python
# HIỆN TẠI — task không được track, lỗi silently dropped:
asyncio.create_task(  # noqa: RUF006
    log_usage_event(pool, ...)
)

# FIX OPTION A (đơn giản nhất cho M2): await trực tiếp — block nhẹ nhưng đảm bảo ghi
# Usage logging không phải hot-path — thêm vài ms hoàn toàn chấp nhận được cho MCP
await log_usage_event(pool, ...)

# FIX OPTION B (giữ non-blocking, thêm error logging):
async def _log_with_error_guard() -> None:
    try:
        await log_usage_event(pool, ...)
    except Exception as exc:  # noqa: BLE001
        logger.error("ask_wiki_usage_log_failed: %s", exc)

task = asyncio.create_task(_log_with_error_guard())
# Nếu cần đảm bảo flush khi shutdown: lưu task vào danh sách tracked
```

**Khuyến nghị M2:** Dùng Option A (`await` trực tiếp) vì MCP tool là read-only, latency không quan trọng bằng correctness. Option B phù hợp hơn nếu `log_usage_event` có thể tốn >1s (LLM provider chậm).

## Warnings

### WR-01: Tham số `pool` trong `authenticate_mcp_request` không được dùng — gây nhầm lẫn caller

**File:** `Hub_All/api/app/mcp/auth.py:44-47`

**Vấn đề:** Signature `authenticate_mcp_request(ctx, pool: asyncpg.Pool)` nhận `pool` nhưng thực tế KHÔNG dùng `pool` ở đâu trong thân hàm — hàm tự lấy engine qua `get_engine()` để tạo `AsyncSession`. Comment ghi `# noqa: ARG001 — reserve param`. Tuy nhiên, đây là **vấn đề misleading API**: caller phải truyền `pool` (và phải gọi `_get_pool()` trước), nhưng pool đó không có tác dụng gì. Nếu `get_engine()` chưa được init (ví dụ `init_engine` chưa gọi), hàm sẽ raise `RuntimeError` từ `get_engine()` thay vì dùng `pool` đã được inject.

Rủi ro thực tế: nếu SQLAlchemy engine fail init nhưng asyncpg pool OK (edge case khi step 6 lifespan fail), MCP auth sẽ fail dù pool đã sẵn sàng — trong khi caller nghĩ pool đang được dùng.

```python
# FIX: Xoá tham số pool nếu thực sự không dùng, hoặc dùng nó thật sự.
# Option A — xoá param (breaking change nhỏ, cập nhật caller):
async def authenticate_mcp_request(ctx: object) -> UserWithHubs:
    ...

# Option B — dùng pool thật sự thay vì get_engine() (nhất quán hơn với Pattern 4):
async def authenticate_mcp_request(ctx: object, pool: asyncpg.Pool) -> UserWithHubs:
    async with pool.acquire() as conn:
        # dùng asyncpg trực tiếp thay vì AsyncSession
        ...
```

### WR-02: `mcp>=1.9.4,<2` quá rộng — minor version break có thể xảy ra

**File:** `Hub_All/api/pyproject.toml:37`

**Vấn đề:** `pyproject.toml` pin `mcp>=1.9.4,<2`, nhưng code trong `server.py` (dòng 15-20) note rõ API đã khác nhau hoàn toàn giữa 1.9.4 và 1.27.1 (`http_app()` không tồn tại, `combine_lifespans` không tồn tại, `stateless_http` đặt khác chỗ). ### WR-03: Thiếu validation `query` rỗng trong `search_wiki` và `ask_wiki`

**File:** `Hub_All/api/app/mcp/server.py:130-135, 196-200`

**Vấn đề:** Cả `search_wiki` và `ask_wiki` đều nhận `query: str` nhưng không validate `query.strip()`. Nếu MCP client truyền `query=""` hoặc `query="   "`, call sẽ đến `SearchService`/`AskService` và:
- `SearchService` sẽ gọi embedder với chuỗi rỗng → `ValueError` từ embedder (tùy provider) → bị catch bởi `except Exception` → ToolError với message không rõ ràng.
- `AskService` tương tự.

So sánh: `SearchService` trong HTTP router đã validate ở Pydantic schema level (validator trên `SearchRequest.query`). Trong MCP, `SearchRequest` được tạo từ `query` param mà không qua layer validation tương đương.

```python
# FIX: Thêm guard ở đầu tool, trước khi gọi service:
async def search_wiki(ctx: Context, query: str, ...) -> SearchResult:
    pool = _get_pool()
    try:
        user = await authenticate_mcp_request(ctx=ctx, pool=pool)
    except ValueError as e:
        raise ToolError(str(e)) from e

    if not query or not query.strip():
        raise ToolError("INVALID_QUERY: query không được rỗng")
    ...
```
