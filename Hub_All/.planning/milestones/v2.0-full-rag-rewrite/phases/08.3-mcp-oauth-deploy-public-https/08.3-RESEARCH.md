# Phase 8.3: MCP OAuth 2.0 + Deploy Public HTTPS — Research

**Researched:** 2026-05-19
**Domain:** OAuth 2.1 Authorization Server trong MCP Python SDK · public HTTPS deploy
**Confidence:** HIGH (năng lực SDK + flow Claude web đã verify) · MEDIUM (deploy/reverse-proxy là Claude's Discretion)

## Summary

Câu hỏi sống-còn của phase — *"MCP Python SDK có làm Authorization Server đầy đủ được không?"* — **đã được trả lời: CÓ**. Package `mcp` (`mcp.server.fastmcp.FastMCP`, bản pin `>=1.27.0,<1.28` của dự án) định nghĩa protocol `OAuthAuthorizationServerProvider` với đầy đủ 9 method của một AS thật: `register_client` (DCR — RFC 7591), `authorize`, `load_authorization_code`, `exchange_authorization_code`, `exchange_refresh_token`, `load_access_token`, `revoke_token`... Khi truyền `auth_server_provider=...` vào constructor `FastMCP`, SDK tự mount toàn bộ route AS qua `create_auth_routes()`: `/.well-known/oauth-authorization-server` (RFC 8414), `/.well-known/oauth-protected-resource` (RFC 9728), `/authorize`, `/token`, `/register`. **D-01 KHẢ THI — KHÔNG cần escalate, KHÔNG cần IdP/gateway riêng.** `[VERIFIED: github.com/modelcontextprotocol/python-sdk provider.py + FastMCP server.py v1.27.0]`

⚠️ **Lưu ý phân biệt 2 thư viện trùng tên "FastMCP":** (a) `mcp.server.fastmcp.FastMCP` — module nằm TRONG package `mcp` chính thức, là cái dự án đang dùng; (b) `gofastmcp.com` FastMCP v3.x — framework bên thứ ba tách riêng, có `OAuthProxy`/`MultiAuth`. **CONTEXT.md D-01 nói "FastMCP" = (a)**. Mọi recommendation dưới đây dựa trên (a) để không vi phạm pin `mcp>=1.27.0,<1.28` (P19). KHÔNG thêm package `fastmcp` (b).

Claude web "Add custom connector" yêu cầu: OAuth 2.1 + PKCE S256 (bắt buộc), discovery qua header `WWW-Authenticate` trên 401 → protected-resource-metadata → authorization-server-metadata, DCR optional-nhưng-Claude-sẽ-dùng nếu có `registration_endpoint`, redirect URI cố định `https://claude.ai/api/mcp/auth_callback`. `mcp` 1.27 cover hết các yêu cầu này natively. Phần "khó" thực sự của phase KHÔNG phải OAuth SDK mà là **3 mảng glue**: (1) bước login HTML form xác thực bằng tài khoản Medinet (D-02); (2) nơi lưu state OAuth (clients/codes/tokens) khi MCP Service cố tình KHÔNG có DB/Redis; (3) vòng đời JWT downstream API Service hết hạn giữa phiên OAuth dài (D-03).

**Primary recommendation:** Implement một lớp `OAuthAuthorizationServerProvider` riêng trong package `mcp_app` (vd `mcp_app/oauth/provider.py`), truyền vào `FastMCP(..., auth_server_provider=..., auth=AuthSettings(...))`. Bước login = một Starlette route HTML form trong `mcp_app` POST credentials → gọi `POST /api/auth/login` của API Service → nhận JWT pair → bind JWT (cả access + refresh) vào OAuth authorization code/token state. State OAuth lưu **SQLite file cục bộ** (`aiosqlite`, một file trong volume) — đủ nhẹ, sống sót restart, không kéo dependency nặng. Downstream JWT hết hạn: lưu `refresh_token` API Service trong cùng record token OAuth, refresh lười khi gặp 401. Deploy: **Caddy reverse proxy** (auto-TLS Let's Encrypt) thêm vào `docker-compose.yml`, hoặc Cloudflare Tunnel nếu không mở được port 80/443 — planner chốt theo hạ tầng thật.

**Trace use-case chính (kết nối connector lần đầu):**
1. Claude web probe MCP URL → MCP Service trả `401` + header `WWW-Authenticate: Bearer resource_metadata="..."`.
2. Claude đọc `/.well-known/oauth-protected-resource` → lấy `authorization_servers` → đọc `/.well-known/oauth-authorization-server`.
3. Claude `POST /register` (DCR) → MCP Service `register_client()` lưu client.
4. Claude redirect user tới `GET /authorize?...code_challenge=...&redirect_uri=https://claude.ai/api/mcp/auth_callback`.
5. MCP Service `authorize()` → redirect tới `GET /login` (HTML form).
6. User nhập username/password Medinet → `POST /login/callback` → MCP Service gọi `POST /api/auth/login` → nhận JWT pair.
7. MCP Service phát authorization code, bind (user, JWT access, JWT refresh) vào code → redirect về `redirect_uri` của Claude kèm `code`.
8. Claude `POST /token` (code + PKCE verifier) → `exchange_authorization_code()` → phát OAuth access token (+ refresh) → token bind tới downstream JWT trong store.
9. Tool call: Claude gửi `POST /mcp` + `Authorization: Bearer <oauth_token>` → SDK `load_access_token()` verify → tool lấy downstream JWT từ store → `ApiClient` forward `Authorization: Bearer <JWT>` xuống API Service.

### Pattern 1: Wire FastMCP as Authorization Server
**What:** Truyền `auth_server_provider` + `AuthSettings` vào `FastMCP(...)`; SDK tự mount route AS.
**When to use:** Bắt buộc cho D-01 — MCP Service đóng vai AS.
**Example:**
```python
# Source: github.com/modelcontextprotocol/python-sdk examples/servers/simple-auth (v1.27.0)
from mcp.server.auth.settings import AuthSettings, ClientRegistrationOptions, RevocationOptions
from mcp.server.fastmcp import FastMCP
from pydantic import AnyHttpUrl

provider = MedinetOAuthProvider(...)  # lớp tự viết, kế thừa OAuthAuthorizationServerProvider

mcp = FastMCP(
    "Medinet Wiki MCP Service",
    stateless_http=True,
    auth_server_provider=provider,            # → SDK mount /authorize /token /register
    auth=AuthSettings(
        issuer_url=AnyHttpUrl(settings.oauth_issuer_url),   # PHẢI là URL public HTTPS
        resource_server_url=AnyHttpUrl(settings.oauth_issuer_url),
        client_registration_options=ClientRegistrationOptions(
            enabled=True,                     # DCR mở cho Claude web (Claude's Discretion)
            valid_scopes=["wiki"],
            default_scopes=["wiki"],
        ),
        revocation_options=RevocationOptions(enabled=True),
        required_scopes=["wiki"],
    ),
)
```
`create_auth_routes()` được SDK gọi bên trong `streamable_http_app()` khi `auth_server_provider` non-None — KHÔNG cần tự thêm route metadata. `[VERIFIED: FastMCP server.py v1.27.0 — "if self._auth_server_provider: routes.extend(create_auth_routes(...))"]`

### Pattern 2: Login form route — verify credential Medinet (D-02)
**What:** Route HTML form custom trong `mcp_app/oauth/login.py`, POST credentials → `POST /api/auth/login` của API Service.
**When to use:** Bước login/consent của OAuth flow.
**Example:**
```python
# Pattern theo simple-auth example: login_page_handler + handle_login_callback
# GET /login  → trả HTML <form method=POST action=/login/callback>
# POST /login/callback:
async def handle_login_callback(request):
    form = await request.form()
    # Gọi API Service xác thực — KHÔNG verify password tại MCP Service (D-02).
    jwt_pair = await api_client.login(form["username"], form["password"])
    # jwt_pair = {access_token, refresh_token, user}
    if jwt_pair is None:
        return render_login_with_error("Sai tài khoản hoặc mật khẩu")
    # Gắn (user, jwt access, jwt refresh) vào authorization code đang chờ.
    auth_code = await provider.complete_authorization(state, jwt_pair)
    return RedirectResponse(f"{redirect_uri}?code={auth_code}&state={oauth_state}")
```
`api_client.login()` là method MỚI — gọi `POST /api/auth/login` body `{email, password}` `[VERIFIED: api/app/auth/router.py — endpoint nhận LoginRequest{email,password}, trả envelope data{access_token, refresh_token, user}]`. **Lưu ý field:** API Service login dùng **`email`** (không phải `username`) — form phải gửi email, hoặc planner quyết label form.

### Pattern 3: Forward JWT downstream + refresh khi hết hạn (D-03)
**What:** Mỗi tool call lấy downstream JWT từ OAuth token state; nếu API trả 401 → dùng API refresh_token đổi JWT mới.
**When to use:** Mọi tool call sau khi đã OAuth.
**Example:**
```python
# api_client._request: đổi header X-API-Key → Authorization Bearer
headers = {"Authorization": f"Bearer {downstream_jwt}"}
# Khi gặp ApiUnauthorizedError lần đầu: thử refresh rồi retry 1 lần.
try:
    return await client.post(path, jwt=jwt, json_body=body)
except ApiUnauthorizedError:
    new_pair = await client.refresh_jwt(downstream_refresh_token)  # POST /api/auth/refresh
    if new_pair is None:
        raise  # refresh token cũng hết hạn → ToolError, user re-connect
    await store.update_downstream_jwt(oauth_token, new_pair)
    return await client.post(path, jwt=new_pair["access_token"], json_body=body)
```
JWT API Service: access TTL **900s (15 phút)**, refresh TTL **604800s (7 ngày)** `[VERIFIED: api/app/auth/jwt.py docstring + issue_token_pair]`. Refresh có **rotation** (token cũ blacklist) `[VERIFIED: REQUIREMENTS.md AUTH-02]` — mỗi lần refresh phải lưu refresh_token mới đè cái cũ, nếu không lần refresh sau sẽ fail.

### Anti-Patterns to Avoid
- **Tự verify password trong MCP Service:** vi phạm D-02 + nhân bản Argon2 logic. Luôn ủy thác `POST /api/auth/login`.
- **Import `app.*` của API Service vào `mcp_app`:** vi phạm ranh giới Phase 8.2 (MCP Service = process độc lập, dependency tối thiểu). Mọi giao tiếp qua HTTP.
- **Bump `mcp` lên ≥1.28 để "có tính năng mới":** vi phạm P19. 1.27.x đã đủ.
- **Dùng in-memory dict cho OAuth state ở prod:** restart MCP Service → Claude phải DCR + login lại. Dùng SQLite file.
- **Log giá trị token/JWT/password:** vi phạm T-08.2-03-I2 (pattern bảo mật hiện có). Chỉ log code/path/status.
- **Để issuer URL = localhost trong prod:** OAuth metadata `issuer` PHẢI là URL public HTTPS Claude truy cập được, nếu không discovery + redirect sẽ sai. Config qua `MCP_OAUTH_ISSUER_URL`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OAuth metadata endpoints (RFC 8414/9728) | Tự viết route `/.well-known/...` | `auth_server_provider` → SDK `create_auth_routes()` | SDK mount đúng spec + đồng bộ format Claude probe |
| `/authorize` + `/token` + PKCE S256 verify | Tự xử lý authorization code + PKCE | `OAuthAuthorizationServerProvider` interface | SDK xử lý PKCE, state, code exchange; chỉ cần implement persistence |
| DCR endpoint `/register` (RFC 7591) | Tự parse client metadata | `ClientRegistrationOptions(enabled=True)` + `register_client()` | SDK validate client metadata, chỉ cần lưu |
| JWT RS256 cho API Service | Tự ký JWT giả danh user | `POST /api/auth/login` | Identity thật phải do API Service phát (D-02/D-03) |
| TLS certificate + renewal | Tự quản cert | Caddy auto-TLS (ACME) hoặc Cloudflare Tunnel | Auto-renew, zero-config; certbot cần cron riêng |
| Password verify | Tự verify Argon2 | API Service `/api/auth/login` | D-02 — KHÔNG nhân bản auth logic |

**Key insight:** Phần OAuth "đúng spec" (metadata, PKCE, code flow, DCR) đã được `mcp` SDK lo trọn — sai sót thường đến từ implement thủ công. Việc của phase này chỉ là **3 hàm persistence** (`store.py`), **1 form login** (`login.py`), và **wiring** (`server.py` + `provider.py`). Đừng viết lại OAuth.

## Common Pitfalls

### Pitfall 1: Nhầm `mcp.server.fastmcp.FastMCP` với `fastmcp` package (gofastmcp.com)
**What goes wrong:** Tài liệu/blog "FastMCP OAuth" thường nói về package `fastmcp` v3.x (bên thứ ba) có `OAuthProxy`. Code theo đó → cài nhầm package, vi phạm pin `mcp`.
**Why it happens:** Hai thứ trùng tên; `fastmcp` từng được merge vào `mcp` rồi lại tách ra phát triển riêng.
**How to avoid:** Chỉ dùng `from mcp.server.fastmcp import FastMCP` và `from mcp.server.auth import ...`. KHÔNG `pip install fastmcp`. Mọi API trong RESEARCH.md này thuộc package `mcp` 1.27.
**Warning signs:** Import `from fastmcp import ...`; xuất hiện `OAuthProxy`/`MultiAuth`/`RemoteAuthProvider` (đó là API của package kia).

### Pitfall 2: `stateless_http=True` xung đột với phiên OAuth?
**What goes wrong:** Lo ngại `stateless_http=True` (8.2) phá OAuth.
**Why it happens:** Nhầm "stateless HTTP session" của MCP transport với "OAuth state".
**How to avoid:** `stateless_http` chỉ nói MCP Streamable transport không giữ session giữa request — KHÔNG liên quan OAuth token. OAuth state nằm ở store riêng (`store.py`). Giữ `stateless_http=True`. Token verification chạy độc lập mỗi request qua `load_access_token()`.
**Warning signs:** Tool call không thấy token sau khi vừa OAuth xong → nghi store, không nghi `stateless_http`.

### Pitfall 3: OAuth state mất khi MCP Service restart
**What goes wrong:** Dùng in-memory dict → restart → Claude báo connector lỗi, phải xóa + add lại connector.
**Why it happens:** Phase 8.2 cố ý cho MCP Service không có DB → cám dỗ dùng dict.
**How to avoid:** SQLite file trong volume mount bền. Schema tối thiểu 3 bảng: `oauth_clients`, `oauth_auth_codes` (TTL ngắn ~10 phút, cleanup), `oauth_tokens` (chứa cả downstream JWT + refresh).
**Warning signs:** Connector hoạt động rồi đột nhiên 401 sau deploy/restart.

### Pitfall 4: Downstream JWT hết hạn giữa phiên OAuth dài
**What goes wrong:** OAuth token sống nhiều ngày; JWT API Service chỉ 15 phút → sau 15 phút mọi tool call 401.
**Why it happens:** Hai vòng đời token độc lập (Claude's Discretion đã nêu rõ).
**How to avoid:** Lưu API `refresh_token` (7 ngày) cùng OAuth token. Tool call gặp 401 → `POST /api/auth/refresh` → đổi JWT mới → lưu refresh mới (rotation!) → retry 1 lần. Nếu refresh cũng fail → ToolError yêu cầu re-connect.
**Warning signs:** Tool chạy được ~15 phút đầu rồi 401 đồng loạt.

### Pitfall 5: Refresh token rotation — lưu thiếu token mới
**What goes wrong:** `POST /api/auth/refresh` rotate (blacklist refresh cũ, phát refresh mới). Nếu chỉ lưu access mới mà giữ refresh cũ → lần refresh kế tiếp dùng refresh đã blacklist → fail.
**Why it happens:** AUTH-02 rotate cả cặp; dễ quên cập nhật refresh.
**How to avoid:** Mỗi lần refresh, ghi đè CẢ `access_token` lẫn `refresh_token` vào record store.
**Warning signs:** Refresh lần 1 OK, lần 2 fail 401.

### Pitfall 6: Issuer URL sai → discovery/redirect hỏng
**What goes wrong:** `issuer_url` để `http://localhost:8190` → metadata trả URL nội bộ → Claude không truy cập được, redirect sai.
**Why it happens:** Dev default vô tình lên prod.
**How to avoid:** `MCP_OAUTH_ISSUER_URL` = đúng domain public HTTPS. Thêm `field_validator` ép `https` cho prod (giống validator `api_base_url` hiện có). Reverse proxy phải forward `X-Forwarded-Proto`/`Host` đúng (Caddy làm tự động).
**Warning signs:** Claude báo "discovery failed" hoặc redirect về localhost.

### Pitfall 7: Reverse proxy không forward đường dẫn `/.well-known/...` ở root
**What goes wrong:** MCP mount dưới path `/mcp` nhưng metadata phải ở root domain (`/.well-known/oauth-authorization-server`). Proxy cấu hình sai prefix → 404 metadata.
**Why it happens:** Claude probe well-known ở root, không dưới `/mcp`.
**How to avoid:** Reverse proxy forward toàn bộ host tới MCP Service (không strip/rewrite path). SDK đã đặt well-known ở root của ASGI app.
**Warning signs:** `curl https://domain/.well-known/oauth-authorization-server` trả 404.

### Gọi `/api/auth/login` từ ApiClient (method mới)
```python
# mcp_app/api_client.py — thêm method
async def login(self, email: str, password: str) -> dict | None:
    """Xác thực credential Medinet qua API Service (D-02). None nếu sai."""
    resp = await self._client.post("/api/auth/login",
                                   json={"email": email, "password": password})
    env = resp.json()
    if resp.status_code == 200 and env.get("success"):
        return env["data"]   # {access_token, refresh_token, user}
    return None              # 401 → credential sai
```

### Đọc Bearer token trong tool (mở rộng auth.py)
```python
# mcp_app/auth.py — thêm, GIỮ extract_api_key
def extract_oauth_token(ctx) -> str | None:
    """Đọc 'Authorization: Bearer <token>' từ MCP Context."""
    try:
        h = ctx.request_context.request.headers.get("authorization", "")
    except (AttributeError, LookupError, TypeError):
        return None
    parts = h.split(" ", 1)
    return parts[1] if len(parts) == 2 and parts[0].lower() == "bearer" else None
```

## Open Questions

1. **Domain public HTTPS cụ thể + ai quản DNS?**
   - What we know: Cần một domain trỏ tới host chạy MCP Service; `MCP_OAUTH_ISSUER_URL` = domain đó.
   - What's unclear: Tên domain, host đặt ở đâu (VPS? on-prem?), có mở được port 80/443 không.
   - Recommendation: Planner ghi là tiền-điều-kiện UAT; người vận hành cung cấp domain trước khi test SC2/SC4. Nếu không mở được port → Cloudflare Tunnel (không cần mở port).

2. **OAuth access token lifetime do MCP Service phát là bao lâu?**
   - What we know: Claude's Discretion. Token OAuth nên đủ dài để không phiền user, có refresh.
   - What's unclear: Con số cụ thể (vài giờ? vài ngày?).
   - Recommendation: Đề xuất access OAuth ~1h + refresh OAuth ~30 ngày; downstream JWT refresh ngầm (Pitfall 4). Planner chốt.

3. **SQLite file đặt ở đâu khi chạy cả native lẫn Docker?**
   - What we know: Cần volume bền cho Docker; native thì file tương đối.
   - What's unclear: Đường dẫn chuẩn.
   - Recommendation: Config `MCP_OAUTH_STATE_DB_PATH`, default tương đối cho native, override tuyệt đối + volume cho Docker (giống cách `COCOINDEX_DB` xử lý ở `docker-compose.yml`).

4. **SC4 (kết nối thật từ Claude web) — UAT thủ công.**
   - What we know: CONTEXT.md `<specifics>` đã xác nhận SC4 là UAT, `human_needed`.
   - Recommendation: Planner đánh dấu SC4 là `human_needed`, không auto-verify; ghi vào `08.3-HUMAN-UAT.md`.

### Known Threat Patterns for MCP OAuth Authorization Server

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Authorization code interception / replay | Spoofing/Tampering | PKCE S256 bắt buộc (SDK); code TTL ngắn + single-use; cleanup code đã dùng |
| Open DCR bị lạm dụng đăng ký client rác | DoS/Tampering | Gate thật là login credential Medinet (D-02 — CONTEXT đã nêu rõ); cân nhắc rate-limit `/register`; code chỉ cấp sau login OK |
| Token theft / không revoke được | Spoofing | Opaque token + store → `revoke_token` hoạt động; `RevocationOptions(enabled=True)` |
| Credential brute-force qua login form | Spoofing | `/api/auth/login` đã có anti-timing oracle (Phase 3); cân nhắc rate-limit route `/login/callback` của MCP Service |
| Downstream JWT leak qua log | Information Disclosure | KHÔNG log Authorization header / JWT (T-08.2-01-I pattern); chỉ log method/path/status |
| `confused deputy` — MCP Service dùng quyền sai user | Elevation of Privilege | Mỗi OAuth token bind đúng 1 downstream JWT của đúng user; KHÔNG dùng service key chung (D-03) |
| Issuer/redirect spoofing | Tampering | `issuer_url` cố định = domain public; PKCE + state; redirect URI khớp client đã đăng ký |
| MITM trên kênh public | Tampering/Disclosure | TLS bắt buộc (Caddy auto-TLS / Cloudflare); HTTP nội bộ container chỉ trong docker network |
