## Decisions Made

- **Test PKCE giới hạn vào phần provider chịu trách nhiệm** — provider glue KHÔNG verify `code_verifier`; verify thật do SDK mcp 1.27 (`TokenHandler.handle` route `/token`) thực hiện (tính `base64url(sha256(code_verifier))` so với `auth_code.code_challenge` → HTTP 400 `invalid_grant` nếu sai — xác nhận bằng inspect). `test_pkce_challenge_persisted` chỉ guard điều provider THỰC SỰ chịu trách nhiệm: `code_challenge` đi nguyên vẹn `oauth_pending` → `AuthorizationCode`. Nếu challenge bị mất/biến dạng ở provider, PKCE của SDK mất đầu vào và vô hiệu — test này là guard chống regress đó. Không còn nhánh "NOVERIFY"/escalate.
- **`test_exchange_code_expired_rejected` seed code trực tiếp qua `oauth_store.save_auth_code`** — `complete_authorization` luôn đặt `expires_at = now + _AUTH_CODE_TTL` nên không tạo được code hết hạn qua flow thường. Test seed thẳng store + dựng tay `AuthorizationCode` để chứng minh guard `expires_at` nằm trong `exchange_authorization_code` (bảo vệ cả đường gọi provider trực tiếp bỏ qua `load_authorization_code`).
- **Fault-injection bằng `patch.object(oauth_store, "save_token", side_effect=_boom)`** — mô phỏng `save_token` lỗi giữa chừng để chứng minh thứ tự `claim_auth_code` (xoá code) TRƯỚC `save_token` đảm bảo code không replay được khi bước save lỗi.
