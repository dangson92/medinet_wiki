# Phase 8.3: MCP OAuth 2.0 + Deploy Public HTTPS - Context

**Gathered:** 2026-05-19
**Status:** Ready for planning

<domain>
## Phase Boundary

Thêm lớp **OAuth 2.0** cho MCP Service (`Hub_All/mcp_service/`, port 8190) và đưa service ra **public qua HTTPS**, để Claude web "Add custom connector" kết nối được tới MeWiki MCP. Hiện MCP Service chỉ chạy HTTP localhost + auth bằng header `X-API-Key` — dialog connector của Claude web chỉ chấp nhận URL public HTTPS + OAuth 2.0, không gửi được custom header.

Kiến trúc đích: **Claude web ─(OAuth)─► MCP Service (public HTTPS) ─(JWT nội bộ)─► API Service.**

Phase này đảo decision **D-05 của Phase 8.1** (OAuth từng bị defer) — nay kéo lên vì Claude web connector bắt buộc OAuth.

**Trong phạm vi:** OAuth 2.0 provider trong MCP Service (authorization server / protected resource metadata + DCR) · login bằng tài khoản Medinet Wiki sẵn có · forward danh tính xuống API Service · Dockerfile + deploy public HTTPS · test phủ OAuth flow.

**Ngoài phạm vi:** write tool / MCP resources qua connector (giữ deferred từ 8.1) · hệ identity mới ngoài tài khoản Medinet · đổi bộ 3 tool read-only hiện có.

<decisions>
## Implementation Decisions

### Chiến lược OAuth provider
- **D-01:** **Tự implement OAuth 2.0 ngay trong MCP Service** dùng khả năng OAuth provider của FastMCP. MCP Service đóng cả vai Authorization Server lẫn Resource Server — tự phát token, expose authorization server metadata / protected resource metadata, hỗ trợ Dynamic Client Registration (DCR) cho Claude web. KHÔNG thêm IdP/gateway riêng — nhất quán triết lý ít-ma-sát của Phase 8.1/8.2, tránh áp lực R3 (anti-pivot fatigue). **Research MUST xác minh** FastMCP hỗ trợ OAuth provider tới đâu (metadata endpoints, DCR, token issuance). Nếu FastMCP thiếu năng lực critical → escalate cho user quyết, KHÔNG tự ý chuyển sang phương án gateway.
- **D-02:** Bước login/consent trong luồng OAuth xác thực bằng **tài khoản Medinet Wiki có sẵn** — username/password của bảng `users` API Service (Phase 3 đã có JWT RS256 login). Token OAuth phát ra gắn với đúng user đã đăng nhập. KHÔNG tạo hệ identity mới, KHÔNG delegate Google.

### Mapping danh tính → hub scope
- **D-03:** MCP Service xác thực xuống API Service bằng **JWT RS256 forward theo từng user**. Khi OAuth login thành công, MCP Service login vào API Service như chính user đó → nhận JWT RS256 → bind JWT vào token/session OAuth. Mỗi tool call forward JWT đó xuống API Service (thay cho `X-API-Key`). API Service `get_api_key_or_jwt` đã chấp nhận JWT — không cần sửa API. Hub isolation enforce API-side, live theo user mỗi call. KHÔNG pre-provision `api_keys` cho user, KHÔNG dùng một service key dùng chung.
- **D-04:** Token OAuth **giữ nguyên quyền theo role** của user: admin → cross-hub mọi hub active; user thường → các hub được gán. Không thêm logic giới hạn đặc biệt cho cổng public — giữ behavior service layer hiện có (3 tool `list_hubs`/`search_wiki`/`ask_wiki` vốn đã read-only).

### Claude's Discretion
- **Deploy public HTTPS:** viết Dockerfile cho `mcp_service/` + chọn reverse proxy TLS (Caddy auto-TLS / nginx+certbot / Traefik / Cloudflare Tunnel) + domain + có đưa vào `docker-compose.yml` chung hay không. Research/planner tự quyết — ưu tiên ít ma sát, ưu tiên auto-TLS. Tham chiếu `api/Dockerfile` (multi-stage uv) làm mẫu.
- **X-API-Key local song song:** sau khi thêm OAuth, MCP Service có giữ chấp nhận `X-API-Key` (cho client local Claude Code/Desktop) song song với OAuth hay chuyển hoàn toàn sang OAuth. Khuyến nghị **giữ song song** để không vỡ client local hiện có từ 8.2 — planner chốt.
- **Xử lý JWT downstream hết hạn:** JWT API Service có thời hạn; MCP Service giữ JWT theo user cho một phiên OAuth dài → khi JWT hết hạn giữa phiên, tool call sẽ 401. Planner PHẢI địa chỉ điểm này — lưu/refresh credential downstream (API Service có `refresh_tokens` từ Phase 3) hoặc cơ chế re-login. Cách cụ thể để Claude quyết.
- Token verification mỗi tool call, lifetime + refresh của token OAuth do MCP Service phát, format JSON-RPC error khi token thiếu/sai/hết hạn.
- **DCR mở:** Claude web yêu cầu Dynamic Client Registration → chấp nhận open DCR. Gate bảo mật thực sự là bước login bằng credential Medinet, không phải client registration. Research xác nhận Claude web cần DCR endpoint nào.

</decisions>

## Existing Code Insights

### Reusable Assets
- FastMCP instance `mcp` (`server.py`) — OAuth provider gắn vào đây; `build_asgi_app()` trả Starlette app để thêm metadata routes.
- `require_api_key` / `extract_api_key` (`auth.py`) — mẫu đọc header từ MCP `Context` → mở rộng để trích + verify OAuth token.
- `ApiClient` (`api_client.py`) — đã hỗ trợ forward header tới API Service; D-03 forward JWT thay vì X-API-Key.
- API Service `get_api_key_or_jwt` — đã chấp nhận JWT RS256, không cần sửa khi MCP forward JWT.

### Established Patterns
- MCP Service là process độc lập, dependency tối thiểu, KHÔNG import `app.*` của API Service — OAuth provider phải nằm trong package `mcp_app`, không kéo phụ thuộc API.
- `Settings` pydantic-settings prefix `MCP_` — config OAuth (issuer URL, signing key path…) thêm qua cơ chế này; có sẵn validator chống misconfiguration.
- Transport Streamable HTTP, `stateless_http=True` — OAuth metadata/authorize/token endpoints sống cạnh `streamable_http_app()`.
- Stack pin Python 3.12, không bump version giữa phase (P19) — dependency OAuth mới phải pin exact.

### Integration Points
- `build_asgi_app()` — nơi compose route OAuth (metadata, authorize, token, DCR) cạnh MCP streamable app.
- `docker-compose.yml` — thêm service `mcp_service` + reverse proxy TLS.
- API Service login endpoint — MCP Service gọi để lấy JWT theo user (D-03).

</code_context>
