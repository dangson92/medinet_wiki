# Phase 8.3: Báo cáo Rà soát Mã nguồn

**Thời điểm rà soát:** 2026-05-19
**Độ sâu:** standard
**Số file rà soát:** 23
**Trạng thái:** issues_found (phát hiện vấn đề)

## Tóm tắt

Phase 8.3 thêm một lớp OAuth 2.0 Authorization Server cho MCP Service kèm hạ tầng
triển khai Docker/Caddy. Kiến trúc tổng thể tốt: dependency injection tường minh,
mọi truy vấn SQLite đều dùng placeholder tham số (không SQL injection), nhật ký
được kiểm soát kỹ — không thấy chỗ nào log token/JWT/code/password thô. Lớp test
phủ khá đầy đủ các đường happy/lỗi.

Tuy vậy rà soát phát hiện **2 vấn đề Nghiêm trọng** và **6 Cảnh báo** tập trung vào
tính đúng đắn OAuth: (1) authorization code KHÔNG bị xoá khi `exchange` thất bại
giữa chừng nên có thể bị tái sử dụng (replay), và (2) refresh token rotation của
OAuth có một lỗ hổng race/double-use vì `update_oauth_token` định vị bản ghi theo
`access_token` đã hết hạn. Ngoài ra `complete_authorization` không kiểm tra hạn
của pending-transaction, `revoke_token` xử lý thiếu nhánh, và một vài chỗ
`KeyError` tiềm ẩn khi payload downstream thiếu field.

Lưu ý: phần "đúng spec PKCE/redirect_uri" do SDK `mcp` 1.27 đảm nhận — review này
giả định SDK hoạt động đúng và chỉ soi phần glue code do dự án viết. Một rủi ro
đáng lưu ý: nếu SDK KHÔNG tự verify PKCE `code_verifier` khi exchange thì hệ thống
mất hoàn toàn bảo vệ PKCE (xem WR-01) — cần xác nhận bằng tay.

## Vấn đề Nghiêm trọng (Critical)

### CR-01: Authorization code không bị xoá khi exchange lỗi giữa chừng — nguy cơ replay

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:201-239`
**Vấn đề:**
`exchange_authorization_code` đọc record của code, gọi `save_token(...)`, RỒI mới
`delete_auth_code(...)`. Nếu `save_token` ném lỗi (ví dụ lỗi SQLite, đĩa đầy, hoặc
khi chạy đồng thời 2 request cùng code), code KHÔNG bị xoá khỏi `oauth_auth_codes`
và vẫn còn hạn — client (hoặc kẻ tấn công chặn được code) có thể gọi lại exchange.
RFC 6749 §10.5 yêu cầu authorization code phải single-use; nếu một code bị dùng
quá một lần, Authorization Server PHẢI thu hồi mọi token đã phát từ code đó.

Ngoài ra `load_auth_code` được gọi LẠI trong `exchange_authorization_code`
(dòng 211) dù SDK đã gọi `load_authorization_code` trước đó — không có khoá nào
chống hai request song song cùng đọc code rồi cùng phát token.

**Fix:** Xoá code TRƯỚC khi phát token (hoặc dùng `DELETE ... RETURNING` để
"claim" code nguyên tử), và chỉ tiếp tục nếu xoá thực sự ảnh hưởng 1 dòng:

```python
# store.py — biến delete thành thao tác claim nguyên tử:
async def claim_auth_code(self, code: str) -> dict | None:
    """Đọc + xoá code trong một bước nguyên tử. None nếu code đã bị dùng."""
    conn = self._require_conn()
    async with conn.execute(
        "DELETE FROM oauth_auth_codes WHERE code = ? RETURNING "
        "client_id, code_payload, downstream_jwt, "
        "downstream_refresh_token, user_payload, expires_at",
        (code,),
    ) as cursor:
        row = await cursor.fetchone()
    await conn.commit()
    if row is None:
        return None
    return { ... }  # map như load_auth_code

# provider.exchange_authorization_code:
record = await self._store.claim_auth_code(authorization_code.code)
if record is None:
    raise OAuthStoreError("Authorization code không tồn tại hoặc đã dùng")
# code đã bị xoá — nếu save_token lỗi, code KHÔNG còn để replay
await self._store.save_token(...)
```

### CR-02: Refresh token rotation định vị bản ghi sai khi access token đã hết hạn — double-spend / mất token

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:257-290` và
`Hub_All/mcp_service/mcp_app/oauth/store.py:306-320`
**Vấn đề:**
`exchange_refresh_token` rotate token bằng `update_oauth_token(old_access_token=
record["access_token"], ...)` — tức `UPDATE oauth_tokens SET ... WHERE
access_token = ?`. Có hai lỗi:

1. **Không single-use refresh token.** `exchange_refresh_token` KHÔNG kiểm tra
   record có còn hợp lệ trước khi rotate, và `UPDATE` là idempotent: nếu hai
   request cùng đem một refresh token cũ, request đầu đổi `access_token` thành
   `new_access_A`; request hai vẫn đọc được record (nó query theo `refresh_token`,
   chưa đổi tại thời điểm đọc) và rotate tiếp. Không có cơ chế phát hiện
   refresh-token reuse (RFC 6749 §10.4 / OAuth 2.1 khuyến nghị: phát hiện reuse
   PHẢI thu hồi cả token family). Test `test_exchange_refresh_token_rotates` chỉ
   kiểm tra đường tuần tự, không phủ trường hợp dùng lại đồng thời.

2. **Mệnh đề WHERE dễ trượt.** `update_oauth_token` cập nhật theo
   `WHERE access_token = old`. Nếu trong khoảng giữa `load_token_by_refresh` và
   `update_oauth_token` có một luồng khác đã rotate (đổi `access_token`), `UPDATE`
   khớp 0 dòng — provider vẫn trả về `OAuthToken(access_token=new_access, ...)`
   cho client mà trong store KHÔNG hề có bản ghi nào mang `new_access`. Token mới
   này sẽ luôn 401 ở `load_access_token`. Người dùng bị "đá ra" im lặng.

**Fix:**
- Rotate + kiểm tra số dòng ảnh hưởng; định vị theo `refresh_token` (giá trị
  caller cầm), không theo `access_token`:

```python
async def rotate_token(
    self, old_refresh_token: str, new_access_token: str,
    new_refresh_token: str, expires_at: int,
) -> bool:
    conn = self._require_conn()
    cursor = await conn.execute(
        "UPDATE oauth_tokens SET access_token = ?, refresh_token = ?, "
        "expires_at = ? WHERE refresh_token = ?",
        (new_access_token, new_refresh_token, expires_at, old_refresh_token),
    )
    await conn.commit()
    return cursor.rowcount == 1  # False nếu refresh token đã bị dùng/rotate
```

```python
# provider.exchange_refresh_token:
ok = await self._store.rotate_token(
    old_refresh_token=refresh_token.token,
    new_access_token=new_access,
    new_refresh_token=new_refresh,
    expires_at=self._now() + self._access_token_ttl,
)
if not ok:
    # refresh token đã bị dùng — reuse detected, nên revoke cả family.
    raise OAuthStoreError("Refresh token đã dùng — vui lòng kết nối lại")
```

## Cảnh báo (Warning)

### WR-01: PKCE `code_verifier` không được verify trong glue code — cần xác nhận SDK đảm nhận

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:180-239`
**Vấn đề:**
Provider lưu `code_challenge` vào `oauth_pending` rồi chép sang `code_payload`
của authorization code, và `load_authorization_code` trả `AuthorizationCode(...,
code_challenge=...)`. Nhưng KHÔNG có chỗ nào trong dự án so khớp
`code_verifier` (client gửi khi exchange) với `code_challenge` đã lưu. Nếu
SDK `mcp` 1.27 KHÔNG tự thực hiện bước verify PKCE S256 này, toàn bộ bảo vệ PKCE
mất tác dụng — kẻ chặn được authorization code có thể đổi lấy token mà không cần
`code_verifier`. Comment ở provider.py:9 khẳng định "SDK lo phần PKCE S256" nhưng
không có test nào của dự án chứng minh điều đó.

**Fix:** Xác nhận bằng tay rằng SDK thực sự verify PKCE (đọc mã
`mcp.server.auth` hoặc thêm một integration test gửi `code_verifier` sai và kỳ
vọng exchange thất bại). Nếu SDK không verify, provider phải tự so khớp
`base64url(sha256(code_verifier)) == code_challenge` trong
`exchange_authorization_code`. Cũng nên lưu ý: `code_challenge_method` luôn bị
hard-code thành `"S256"` ở `authorize()` (dòng 121) — nếu client gửi `plain`,
giá trị thật bị bỏ qua một cách âm thầm; nên reject thay vì ghi đè.

### WR-02: `complete_authorization` không kiểm tra hạn của pending transaction

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:132-176`
**Vấn đề:**
`load_pending(txn)` chỉ trả `None` khi bản ghi không tồn tại — KHÔNG so sánh
`created_at` với thời điểm hiện tại. `cleanup_expired` chỉ xoá pending cũ >600s
khi được gọi, nhưng KHÔNG có scheduler nào gọi `cleanup_expired` định kỳ (xem
WR-06). Hệ quả: một `txn` được tạo nhiều giờ/ngày trước vẫn dùng được để hoàn tất
login — cửa sổ tấn công cho một txn bị lộ rộng hơn thiết kế (login form
nhúng `txn` trong HTML; nếu URL `/login?txn=...` bị lưu lịch sử/log proxy, txn
sống vô thời hạn).

**Fix:** Trong `complete_authorization`, sau khi `load_pending`, kiểm tra tuổi:

```python
pending = await self._store.load_pending(txn)
if pending is None or pending["created_at"] < self._now() - _PENDING_TTL:
    raise OAuthStoreError("Phiên authorize hết hạn hoặc không hợp lệ")
```

(`_PENDING_TTL = 600`, khớp với mốc 600s trong `cleanup_expired`.)

### WR-03: `revoke_token` chỉ xoá access token khi nhận `RefreshToken` có record — bỏ sót và không có RFC fallback

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:314-326`
**Vấn đề:**
Khi SDK truyền `RefreshToken`, code làm `load_token_by_refresh` rồi
`delete_token(record["access_token"])`. Nếu `record is None` (refresh token
không khớp — ví dụ token đã rotate hoặc client gửi token sai), method im lặng
không làm gì và vẫn log "revoke token" thành công. RFC 7009 cho phép trả 200 khi
token không tồn tại, nên không phải lỗi spec, nhưng nhánh `else` (xử lý
`AccessToken`) gọi `delete_token(token.token)` mà KHÔNG xác nhận nó là access
token — nếu SDK truyền vào một object khác `RefreshToken`/`AccessToken` (phòng
vệ), code coi như access token. Quan trọng hơn: revoke theo `access_token` của
record nghĩa là nếu client revoke bằng *refresh* token sau khi access token đã
được rotate, `record["access_token"]` là access token MỚI nhất nên vẫn đúng —
nhưng nếu revoke bằng *access* token CŨ (đã rotate), `delete_token` khớp 0 dòng
và refresh token vẫn sống. Token family không được dọn nhất quán.

**Fix:** Chuẩn hoá: luôn tra cứu record (theo cả access lẫn refresh) rồi xoá
theo `access_token` của record tìm được:

```python
async def revoke_token(self, token: AccessToken | RefreshToken) -> None:
    if isinstance(token, RefreshToken):
        record = await self._store.load_token_by_refresh(token.token)
    else:
        record = await self._store.load_token(token.token)
    if record is not None:
        await self._store.delete_token(record["access_token"])
    logger.info("MedinetOAuthProvider revoke token, client_id=%s", token.client_id)
```

### WR-04: `KeyError` chưa xử lý khi payload downstream thiếu field

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:166-168` và
`Hub_All/mcp_service/mcp_app/server.py:224-233`
**Vấn đề:**
`complete_authorization` truy cập `login_data["access_token"]`,
`login_data["refresh_token"]`, `login_data["user"]` trực tiếp. `login_data` là
`envelope.get("data")` trả từ `api_client.login()` — nếu API Service đổi shape
response, trả `data` thiếu field, hoặc `data` là `None` mà `success` vẫn `True`,
sẽ ném `KeyError`/`TypeError` không được bắt. Lỗi này nổi lên qua `login_callback`
nhưng `login.py` chỉ bắt `ApiClientError` và `OAuthStoreError` — `KeyError` lọt
ra ngoài thành HTTP 500 với stack trace, vi phạm yêu cầu T-08.3-08 "lỗi hạ tầng
render thông báo chung, KHÔNG lộ stack trace".

Tương tự ở `server.py:_call_api`, `new_pair["access_token"]` và
`new_pair["refresh_token"]` được truy cập trực tiếp sau `refresh_jwt`.

**Fix:** Validate shape trước khi dùng, hoặc dùng `.get()` + raise lỗi có kiểm
soát:

```python
required = ("access_token", "refresh_token", "user")
if not all(login_data.get(k) for k in required):
    raise OAuthStoreError("Phản hồi login từ API Service thiếu trường bắt buộc")
```

và trong `login_callback` bổ sung `except (KeyError, TypeError)` để render lỗi
hệ thống chung.

### WR-05: `oauth_tokens.refresh_token` không có ràng buộc UNIQUE — `load_token_by_refresh` có thể nhập nhằng

**File:** `Hub_All/mcp_service/mcp_app/oauth/store.py:55-66, 280-288`
**Vấn đề:**
Bảng `oauth_tokens` đặt `access_token` làm PRIMARY KEY nhưng `refresh_token` chỉ
là cột `TEXT` thường — không UNIQUE, không index. `_new_opaque()` dùng
`secrets.token_urlsafe(32)` nên va chạm thực tế gần như bằng 0, nhưng về mặt
chính xác: nếu vì bug nào đó hai record có cùng `refresh_token`,
`load_token_by_refresh` (`SELECT ... WHERE refresh_token = ?` rồi `fetchone()`)
trả một record bất kỳ — sai im lặng. Thiếu index cũng làm tra cứu refresh token
chậm tuyến tính khi bảng lớn (ngoài scope hiệu năng v1, nhưng UNIQUE thì thuộc
tính đúng đắn).

**Fix:** Thêm `UNIQUE` cho cột `refresh_token` trong `CREATE TABLE` (cho phép
nhiều `NULL` — SQLite coi NULL là khác nhau):

```sql
refresh_token TEXT UNIQUE,
```

### WR-06: `cleanup_expired` được định nghĩa nhưng không bao giờ được gọi — code/pending hết hạn tích tụ

**File:** `Hub_All/mcp_service/mcp_app/oauth/store.py:392-402`,
`Hub_All/mcp_service/mcp_app/server.py` (toàn file)
**Vấn đề:**
`cleanup_expired` dọn auth code hết hạn và pending bỏ dở, nhưng `grep` toàn bộ
`mcp_app/` không thấy lời gọi nào (chỉ test gọi). Không có background task,
scheduler, hay lifespan hook nào kích hoạt nó. Hệ quả:
- `oauth_auth_codes` và `oauth_pending` phình vô hạn (code/pending hết hạn vẫn
  nằm trong DB).
- Quan trọng hơn về bảo mật: `load_authorization_code` có kiểm tra
  `expires_at < now` nên code hết hạn không dùng được — tốt; nhưng pending KHÔNG
  có kiểm tra hạn ở runtime (WR-02), nên pending cũ tồn tại vô thời hạn lại còn
  dùng được.

**Fix:** Gọi `cleanup_expired` định kỳ — đơn giản nhất là một asyncio task khởi
động trong lifespan `_composed_lifespan` của `build_asgi_app`:

```python
async def _cleanup_loop(store: OAuthStore) -> None:
    while True:
        await asyncio.sleep(600)
        try:
            await store.cleanup_expired(int(time.time()))
        except Exception:
            logger.warning("cleanup_expired thất bại")
```

khởi động task này khi startup, cancel khi shutdown.

### IN-03: Login form không có bảo vệ CSRF / rate limit

**File:** `Hub_All/mcp_service/mcp_app/oauth/login.py:93-134`
**Vấn đề:** `POST /login/callback` nhận `email`/`password`/`txn` từ form mà
không có CSRF token và không có giới hạn số lần thử. Vì đây là bước nhập
credential, kẻ tấn công có thể brute-force password qua endpoint này (rate limit
thực tế phụ thuộc API Service `/api/auth/login`). `txn` đóng vai trò gần như
state nhưng không ràng buộc session người dùng. Đây là chấp nhận được ở mức MVP
nội bộ, nhưng nên ghi nhận rủi ro.
**Fix:** Cân nhắc dựa vào rate-limit của API Service (xác nhận `/api/auth/login`
có throttle), hoặc thêm giới hạn ở MCP Service. Ghi rủi ro này vào tài liệu UAT.

### IN-04: `RefreshToken` trả `expires_at=None` — refresh token không có hạn

**File:** `Hub_All/mcp_service/mcp_app/oauth/provider.py:250-255`
**Vấn đề:** `load_refresh_token` trả `RefreshToken(..., expires_at=None)`. Config
có `oauth_refresh_token_ttl` (30 ngày) nhưng giá trị này KHÔNG được dùng ở đâu cả
— `save_token` chỉ lưu `expires_at` cho access token, refresh token thực tế sống
vĩnh viễn cho tới khi rotate hoặc revoke. Field config `oauth_refresh_token_ttl`
trở thành dead config gây hiểu nhầm.
**Fix:** Hoặc thực sự áp TTL cho refresh token (lưu `refresh_expires_at` riêng,
kiểm trong `load_refresh_token` / `exchange_refresh_token`), hoặc xoá field
config `oauth_refresh_token_ttl` và `_refresh_token_ttl` nếu cố ý không giới hạn.

### IN-05: `location` redirect ghép chuỗi thủ công — `client_state` không được URL-encode

**File:** `Hub_All/mcp_service/mcp_app/oauth/login.py:130-132`
**Vấn đề:** `location = f"{redirect_uri}?code={code}"` rồi
`location += f"&state={client_state}"`. `code` từ `secrets.token_urlsafe` nên
an toàn URL, nhưng `client_state` do client OAuth cung cấp tự do — nếu chứa ký tự
`&`, `#`, hay khoảng trắng, URL redirect bị méo (state bị cắt hoặc tạo tham số
giả). Cũng giả định `redirect_uri` chưa có sẵn query string (`?`); nếu có thì URL
hỏng.
**Fix:** Dùng `urllib.parse` để ghép query an toàn:

```python
from urllib.parse import urlencode, urlsplit, urlunsplit, parse_qsl

parts = urlsplit(redirect_uri)
query = dict(parse_qsl(parts.query))
query["code"] = code
if client_state is not None:
    query["state"] = client_state
location = urlunsplit(parts._replace(query=urlencode(query)))
```

## Ghi chú tích cực

- Mọi truy vấn SQLite (`store.py`) đều dùng placeholder `?` tham số hoá — không
  có SQL injection dù `client_id`/`code`/`txn` do client cung cấp. `_TOKEN_COLUMNS`
  được dùng trong f-string nhưng là hằng số nội bộ, không phải dữ liệu người
  dùng — an toàn.
- Không phát hiện chỗ nào log token/JWT/code/password/email thô; nhật ký chỉ ghi
  `client_id`, loại thao tác, `status_code`, `error.code` — đúng T-08.3-08.
- `login.py:render_login_form` escape `txn` và `error` bằng `html.escape` — chống
  XSS phản chiếu trong form.
- `config.py` validate scheme `http`/`https` cho cả `api_base_url` và
  `oauth_issuer_url` — chống SSRF/misconfiguration.
- Dockerfile chạy bằng user không phải root (`mcpuser`), `.dockerignore` loại
  `.env`/`*.db`/`.oauth` khỏi build context — không rò rỉ secret/state vào image.
- `extract_oauth_token` phân tích header `Authorization` chặt chẽ (kiểm `bearer`
  không phân biệt hoa thường, token sau prefix phải non-rỗng).
