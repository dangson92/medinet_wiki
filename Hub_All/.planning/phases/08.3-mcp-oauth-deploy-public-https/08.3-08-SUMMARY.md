### Task 1: store.py — schema family_id + save_token + rotate_token preserve + delete_token_family (commit `dfcec01`)

Vá schema OAuthStore cho HIGH-02 (audit 2026-05-21, RFC 6749 §10.4):

- **Schema:** `_SCHEMA_STATEMENTS` oauth_tokens thêm cột `family_id TEXT` (NULLABLE) SAU `user_payload` và TRƯỚC `expires_at` — khớp thứ tự INSERT/SELECT. NULLABLE để back-compat DB cũ (token Phase 8.3-01..07 chưa có cột này) — token cũ NULL coi như family riêng, không xoá nhầm.
- **Migration back-compat:** `init_schema` thêm khối `PRAGMA table_info(oauth_tokens)` check `family_id` không có → `ALTER TABLE oauth_tokens ADD COLUMN family_id TEXT`. Pattern lặp lại từ Plan 07 (csrf_token). KHÔNG cần migration step thủ công — pull Plan 08 + restart MCP Service là đủ.
- **save_token:** signature thêm tham số `family_id: str | None = None` SAU `user_payload` và `expires_at: int = 0` (default 0 chỉ để cho phép default sau dấu phẩy `family_id` — caller production LUÔN truyền kwarg non-zero). INSERT có 9 cột + 9 placeholder + 9 giá trị.
- **_TOKEN_COLUMNS + _row_to_token:** thêm `family_id` ở vị trí thứ 8 (TRƯỚC expires_at — khớp thứ tự schema). `_row_to_token` trả dict có khoá `family_id`.
- **rotate_token:** KHÔNG đụng body (UPDATE chỉ set 3 cột access/refresh/expires_at → family_id preserve tự động). Docstring cập nhật rõ guarantee preserve + tham chiếu delete_token_family.
- **delete_token_family:** method MỚI sau `delete_token` — `DELETE FROM oauth_tokens WHERE family_id = ?` trả `cursor.rowcount`. Guard `if not family_id: return 0` để no-op với NULL/rỗng (back-compat token cũ).

Acceptance criteria 100% PASS: schema có `family_id TEXT` (2 occurrence — CREATE TABLE + INSERT columns); init_schema có `PRAGMA table_info(oauth_tokens)` + `ALTER TABLE oauth_tokens ADD COLUMN family_id TEXT`; save_token signature có `family_id: str | None = None`; INSERT 9 placeholder; `_TOKEN_COLUMNS` chứa `family_id`; `_row_to_token` trả dict có key `family_id`; `delete_token_family` chứa `DELETE FROM oauth_tokens WHERE family_id = ?` + guard `if not family_id: return 0`; ruff sạch; `test_oauth_store.py` 15/15 PASS (regression — default family_id=None back-compat).

### Task 2: provider.py — sinh family_id + delete_token_family khi reuse (commit `4ef2c99`)

Vá HIGH-02 (audit 2026-05-21, RFC 6749 §10.4) — bản fix CR-02 Plan 05 chỉ chặn double-spend cùng refresh token, nhưng phiên bị đánh cắp (attacker rotate chain mới trước nạn nhân) vẫn sống. Bây giờ thu hồi cả token family khi reuse phát hiện.

provider.py 2 thay đổi:

- **`exchange_authorization_code`:** sinh `family_id = self._new_opaque()` (>160-bit entropy) lúc phát token đầu chain + truyền vào `save_token` kwarg `family_id=family_id`. KHÔNG đụng guard Plan 07 (client_id mismatch sau claim_auth_code) — chỉ thêm 2 dòng sau `refresh_token = self._new_opaque()`. Code sau guard Plan 07 GIỮ NGUYÊN nguyên vẹn.

- **`exchange_refresh_token` nhánh `if not ok:`** (sau rotate_token rowcount mismatch — reuse detected):
  - Load `family_id` từ `record.get("family_id") or ""` (record đã load ở đầu hàm qua `load_token_by_refresh` — sau Task 1, record có khoá `family_id` vì `_row_to_token` expose).
  - Non-NULL → gọi `self._store.delete_token_family(family_id)` xoá cả chain (kẻ tấn công + nạn nhân đều mất phiên, đúng spec §10.4) + log `warning` với `client_id` + `family_size=deleted_rowcount`.
  - NULL/rỗng (token cũ trước migration) → log info back-compat, KHÔNG thu hồi family.
  - Raise `OAuthStoreError("Refresh token đã dùng — vui lòng kết nối lại")` GIỮ NGUYÊN (đảm bảo client biết phải kết nối lại).
  - Guard Plan 07 (record.client_id != client.client_id Ở TRÊN) GIỮ NGUYÊN — CHỈ thay nhánh sau rotate_token.

**Log policy:** chỉ ghi `client_id` + `family_size=N` (số dòng xoá) — KHÔNG log `family_id` giá trị thô (T-08.3-08).

Acceptance criteria 100% PASS: `exchange_authorization_code` chứa `family_id = self._new_opaque()` + `save_token` call có kwarg `family_id=family_id`; `exchange_refresh_token` đoạn `if not ok:` chứa `delete_token_family(family_id)` + `record.get("family_id")`; log warning chứa chuỗi `xoá token family`; ruff sạch; `test_oauth_provider.py` 17/17 PASS (regression — test reuse cũ thấy family bị xoá, hành vi đúng đắn hơn).

### Task 3: server.py — asyncio.Lock per access_token cho refresh JWT + wire cleanup (commit `5469ddd`)

Vá HIGH-03 + đóng Blocker #4 cleanup follow-up (audit 2026-05-21):

**Trước fix HIGH-03:** 2 tool song song cùng OAuth token + downstream JWT vừa hết hạn — tool 1 refresh OK (refresh_token rotate), tool 2 refresh với refresh_token đã rotate ở tool 1 → API 401 → tool 2 fail "MCP_UNAUTHORIZED" không deterministic. User thấy lỗi gián đoạn ngẫu nhiên.

**Sau fix:** single-flight refresh per access_token + re-check trong critical section.

server.py 3 thay đổi:

- **import asyncio top-level** (mới — bổ sung trên `import atexit` ở đầu file). KHÔNG xoá `import asyncio` late trong `_close_client_atexit:439` (giữ scope hẹp — không refactor pre-existing pattern).

- **Lock pool module-level + 2 helper** đặt ngay TRÊN `_call_api`:
  - `_refresh_locks: dict[str, asyncio.Lock] = {}` — pool key=oauth_token, value=Lock.
  - `_get_refresh_lock(oauth_token) -> asyncio.Lock` — lazy create entry nếu chưa có.
  - `_release_refresh_lock(oauth_token) -> None` — `dict.pop` best-effort, KHÔNG raise nếu key không tồn tại. Memory profile: 1 entry ~ 136 byte, 1000 user concurrent ~ 136 KB — chấp nhận được.

- **`_call_api` khối refresh JWT bọc trong `async with _get_refresh_lock(oauth_token)`** — logic chính của HIGH-03 fix:
  - Vào lock → re-load `record = await store.load_token(oauth_token)` để đọc state fresh.
  - **Re-check `record["downstream_jwt"] != cred.value`** — tool 1 đã refresh + lưu store; tool 2 vào lock thấy JWT mới → log info "đã được tool song song refresh" → `return await _do(record["downstream_jwt"])` (skip refresh, retry với JWT mới luôn).
  - Nếu vẫn cùng JWT cũ → tool đầu tiên vào lock thật → gọi `client.refresh_jwt(record["downstream_refresh_token"])` → validate shape new_pair → `store.update_downstream_jwt` (Pitfall 5 — lưu CẢ access + refresh) → retry với JWT mới.

provider.py 1 thay đổi (Blocker #4 — cleanup follow-up):

- **`revoke_token` SAU `delete_token` thành công:**
  - Lưu `access_token_revoked = record["access_token"]` (giữ reference trước khi delete).
  - `await self._store.delete_token(access_token_revoked)`.
  - Late import `from mcp_app.server import _release_refresh_lock` (chống circular — verified bằng grep: 0 top-level `from mcp_app.server` trong provider.py; pattern khớp `_close_client_atexit:439 import asyncio`).
  - Gọi `_release_refresh_lock(access_token_revoked)` → pop entry pool, tránh memory leak ~136 byte / user khi revoke.
  - Guard Plan 07 (record.client_id != token.client_id silent no-op) GIỮ NGUYÊN.

**Threat T-08.3-40 (DoS — refresh JWT race) + T-08.3-43 (Info Disclosure — lock leak revoke)** mitigated.

Acceptance criteria 100% PASS: `_refresh_locks: dict[str, asyncio.Lock] = {}` xuất hiện 1 lần ở module-level + 3 reference khác (helper body); `_get_refresh_lock` 3 occurrence (def + 2 usage); `_release_refresh_lock` 2 occurrence ở server.py (def + nothing else, vì caller late import từ provider.py — 2 occurrence trong provider.py); `async with _get_refresh_lock(oauth_token):` xuất hiện trong `_call_api`; re-check `record["downstream_jwt"] != cred.value` trong khối refresh; `import asyncio` top-level; `provider.revoke_token` chứa late import + call `_release_refresh_lock(access_token_revoked)`; grep verify 0 top-level `from mcp_app.server` trong provider.py; ruff sạch; `pytest -q` 116/116 PASS; `pytest -q -m critical` 17/17 PASS.
