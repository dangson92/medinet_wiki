### Task 1: provider.py — client_id mismatch + DCR redirect_uri whitelist (commit `d9256cb`)

Vá 1 Critical + 3 High security ở một file:

- **HIGH-01:** Thêm `_ALLOWED_REDIRECT_HOSTS` + `_ALLOWED_REDIRECT_HOST_SUFFIXES` + helper `_validate_redirect_uri(uri)` validate scheme/host theo whitelist (claude.ai, modelcontextprotocol.io, localhost, 127.0.0.1; http chỉ cho loopback). `register_client` loop validate mọi `redirect_uri` của `client_info` TRƯỚC `save_client` → chặn code interception attack qua DCR (attacker đăng ký redirect_uri trỏ về domain mình).
- **CRIT-03 sub-fix 2:** `load_authorization_code` kiểm `record["client_id"] != client.client_id` → return None (giữ semantics "code không tồn tại", không leak thông tin attacker đoán đúng code).
- **CRIT-03 sub-fix 1 + HIGH-08:** `exchange_authorization_code` kiểm mismatch SAU `claim_auth_code` (code đã bị xoá nguyên tử) → raise `OAuthStoreError("Authorization code không thuộc client này")`. Đóng chain attack với `_BasicAuthFormShim` (HIGH-08 hết ý nghĩa khai thác sau khi vá CRIT-03 — shim inject client_id từ Basic header, mismatch check chặn cross-client).
- **HIGH-08 family:** `exchange_refresh_token` kiểm `record["client_id"] != client.client_id` defense-in-depth (RFC 6749 §4.1.3 / §6).
- **HIGH-07:** `revoke_token` kiểm `record["client_id"] != token.client_id` → silent no-op (RFC 7009 §2.2 — "invalid tokens do not cause an error response"). KHÔNG xoá, KHÔNG raise.

Log chỉ ghi `client_id` + tên thao tác + `mismatch` — KHÔNG log giá trị token/code/JWT thô (T-08.3-08).

Acceptance criteria 100% PASS: `record["client_id"] != client.client_id` xuất hiện 3 lần (load + exchange_code + exchange_refresh), `record["client_id"] != token.client_id` 1 lần (revoke), `_validate_redirect_uri` + `_ALLOWED_REDIRECT_HOSTS` + `for redirect_uri in client_info.redirect_uris` đầy đủ; ruff sạch; `test_oauth_provider.py` 17/17 PASS (regression, test cũ dùng cùng client trong load + exchange nên mismatch không trigger).
