## I. Critical — chặn ship/UAT

### CRIT-01 — CORS `allow_origins=*` cho cả endpoint nhạy cảm
**File:** [mcp_service/mcp_app/server.py:769-776](../../../mcp_service/mcp_app/server.py#L769-L776)
**Trục:** B (OAuth & security)

`_add_cors_middleware` set `allow_origins=["*"]` + `allow_headers=["*"]` + `expose_headers=["Mcp-Session-Id", "WWW-Authenticate"]`. Áp dụng cho tất cả route gồm `/token`, `/authorize`, `/revoke`, `/register`, và transport `/mcp`. RFC 8414 §3.1 / RFC 9728 §3.1 chỉ yêu cầu CORS cho **metadata endpoint**, KHÔNG cho token endpoint.

**Khai thác:** Malicious browser extension hoặc tab compromised có thể gọi `/token` + transport tools từ origin bất kỳ chỉ cần biết Bearer token (vd qua XSS app khác). Phòng vệ duy nhất là token nằm trong memory client OAuth — Claude/Inspector đều giữ client-side, có thể leak qua XSS.

**Fix:**
```python
# Tách 2 CORS policy:
# (1) Metadata-only: allow_origins=["*"] cho /.well-known/*
# (2) Sensitive: whitelist origin cho /token, /authorize, /revoke, /register, /mcp
allowed_origins_sensitive = [
    "https://claude.ai",
    "https://inspector.modelcontextprotocol.io",
    # dev: f"http://localhost:6274"
]
```
Cần 2 instance CORSMiddleware riêng (Starlette không support per-route CORS native — dùng 2 Starlette sub-app).

### CRIT-02 — CORS middleware add 2 lần (path-prefix mode)
**File:** [mcp_service/mcp_app/server.py:607,702](../../../mcp_service/mcp_app/server.py#L607)
**Trục:** B (OAuth)

Path-prefix mode (production deploy chính theo `MCP_PATH_PREFIX=mcp`):
- Dòng 607: `_add_cors_middleware(inner)` — gắn lên inner Starlette app.
- Dòng 702: `_add_cors_middleware(wrapper)` — gắn lên wrapper Starlette.

Request `POST /mcp/token` đi qua wrapper Router → match `Mount("/mcp", inner)` → forward xuống inner Router → match `/token`. Cả 2 CORSMiddleware đều inject header → response có `Access-Control-Allow-Origin: *` **2 lần**.

**Hậu quả:** CORS spec (Fetch §3.1) quy: header value MUST be single value hoặc list comma-separated, KHÔNG duplicate header name. Browser hiện đại reject response — toàn bộ OAuth flow vỡ với Claude web / MCP Inspector. Hiện test `test_cors_actual_get_metadata` PASS vì test gọi metadata ở root path (chặn bởi wrapper, không qua inner).

**Fix:** Chỉ add CORS ở wrapper outer trong path-prefix mode (xoá dòng 607). Subdomain mode giữ nguyên ở inner.

**Test thêm:** `test_cors_no_duplicate_in_path_prefix_mode` — set `MCP_PATH_PREFIX=mcp`, GET `/mcp/.well-known/...` → assert response header `Access-Control-Allow-Origin` xuất hiện ĐÚNG 1 lần (`response.headers.getlist("access-control-allow-origin")` len == 1).

### CRIT-03 — Chain client_id mismatch + BasicAuthFormShim → cross-client code exchange
**File:** [mcp_service/mcp_app/oauth/provider.py:296-342](../../../mcp_service/mcp_app/oauth/provider.py#L296-L342), [provider.py:275-294](../../../mcp_service/mcp_app/oauth/provider.py#L275-L294), [server.py:779-961](../../../mcp_service/mcp_app/server.py#L779)
**Trục:** B (OAuth) — chain A-05 + A-06 + B-07

`exchange_authorization_code` (dòng 296) và `load_authorization_code` (dòng 275) đều **KHÔNG kiểm `record["client_id"] == client.client_id`**. Đây là IN-04 đã ghi nhận trong `08.3-REVIEW.md` (2026-05-19) — vẫn còn.

Chain khai thác:
1. Victim flow normal: `client_id=victim, code=ABC` lưu trong `oauth_auth_codes`.
2. Attacker đăng ký DCR riêng (`client_id=attacker, client_secret=X`).
3. Attacker đoán/biết `code=ABC` (leak qua referrer header, log proxy, hoặc redirect_uri attacker-controlled — xem CRIT-05/B-02 chưa fix).
4. Attacker POST `/mcp/token` với:
   - `Authorization: Basic base64("attacker:X")` HOẶC form body `client_id=attacker&client_secret=X`.
   - Form `grant_type=authorization_code&code=ABC&code_verifier=<attacker's>`.
5. SDK ClientAuthenticator verify `attacker:X` → PASS (đúng credentials của attacker).
6. SDK gọi `load_authorization_code(client=attacker, "ABC")` → trả `AuthorizationCode(client_id=victim, ...)` → SDK KHÔNG kiểm mismatch.
7. SDK gọi `exchange_authorization_code(client=attacker, AuthorizationCode(...))` → provider claim code + save_token với `record["client_id"]=victim, downstream_jwt=victim's JWT`.
8. Attacker nhận access_token bind downstream JWT của victim → có thể gọi tools với danh tính victim.

PKCE bảo vệ một phần nếu `code_verifier` không leak — nhưng nếu attacker phát hiện được verifier (vd qua localhost port collision Inspector v0.21.2, malicious extension), chain chạy đầy đủ.

**Fix (5 dòng):**
```python
# provider.py:307 trong exchange_authorization_code, sau claim_auth_code:
if record["client_id"] != client.client_id:
    raise OAuthStoreError("Authorization code không thuộc client này")
```
Và tương tự trong `load_authorization_code` dòng 280, ngay sau load_auth_code.

**Test thêm:** `test_exchange_code_rejects_client_id_mismatch` — save_auth_code với client_id=A, gọi `exchange_authorization_code(client=B, ...)` → `pytest.raises(OAuthStoreError)`.

### HIGH-01 — DCR `redirect_uris` không validate
**File:** [provider.py:159-168](../../../mcp_service/mcp_app/oauth/provider.py#L159-L168)
**Trục:** B

`register_client` (DCR RFC 7591) lưu thẳng `redirect_uris` của client_info. Docstring `provider.py:18-19` ghi rõ "DCR mở (D-01 Claude's Discretion) — gate bảo mật thật là bước login credential Medinet". Vấn đề: attacker đăng ký client với `redirect_uri=https://attacker.com/cb` → user login form ở MCP → PROVIDER redirect code về attacker domain.

PKCE bảo vệ nếu code_verifier không leak (xem CRIT-03), nhưng đây vẫn là vector code interception. Nếu user login bằng tài khoản Medinet thật → attacker giữ code → có PKCE verifier của attacker (đã gửi từ đầu) → exchange thành công.

**Fix:** Ép `redirect_uri.scheme ∈ {"https", "http"}` và `redirect_uri.host` whitelist:
- `*.claude.ai` (Claude web)
- `*.modelcontextprotocol.io` (Inspector official)
- `localhost`, `127.0.0.1` (dev)

Hoặc tối thiểu: HTTPS-only ngoài localhost.

### HIGH-02 — Refresh token family revocation thiếu (WR-01 vẫn còn)
**File:** [provider.py:382-390](../../../mcp_service/mcp_app/oauth/provider.py#L382-L390)
**Trục:** B
**Khớp finding `08.3-REVIEW.md` WR-01 — chưa fix.**

`exchange_refresh_token` phát hiện reuse (rotate_token trả `False`) → raise `OAuthStoreError` cho request hiện tại NHƯNG refresh token mới (vừa được phát ra trước đó cho lần rotate thành công) VẪN HOẠT ĐỘNG. RFC 6749 §10.4 khuyến nghị mạnh: phát hiện reuse → AS PHẢI thu hồi toàn bộ token family.

**Khai thác:** Attacker đoạt refresh token cũ + rotate trước victim → attacker giữ refresh mới hoạt động. Victim sau đó reuse refresh cũ → phát hiện reuse → victim bị 401 nhưng attacker vẫn truy cập được.

**Fix:** Thêm column `family_id UUID` trong `oauth_tokens`, gán cố định lúc `exchange_authorization_code`, giữ nguyên qua mỗi `rotate_token`. Khi rowcount mismatch:
```python
await self._store.delete_token_family(record["family_id"])
```

### HIGH-03 — Refresh JWT concurrency race (no asyncio.Lock)
**File:** [server.py:218-243](../../../mcp_service/mcp_app/server.py#L218-L243)
**Trục:** A (code bugs)

2 tool call song song dùng cùng OAuth token + downstream JWT vừa hết hạn:
1. Tool 1 nhận 401 → load record → gọi `refresh_jwt(refresh_token_old)` → API trả pair mới → lưu store.
2. Tool 2 (song song) load record (CHƯA update) → gọi `refresh_jwt(refresh_token_old)` → API trả 401 (refresh token đã rotate ở Tool 1) → tool fail "Phản hồi refresh JWT thiếu trường bắt buộc" hoặc re-raise 401.

**Hệ quả:** Tool fail không deterministic khi concurrent. User thấy "MCP_UNAUTHORIZED" gián đoạn.

**Fix:** `asyncio.Lock` per OAuth access_token. Hoặc dùng database-level lock SQLite (`BEGIN IMMEDIATE`) qua dedicated store method `acquire_refresh_lock(oauth_token)` + retry-once.

**Test thêm:** `test_refresh_jwt_concurrency` — `asyncio.gather` 2 tool call với JWT 401, assert chỉ 1 lần `client.refresh_jwt` được gọi và cả 2 tool succeed.
