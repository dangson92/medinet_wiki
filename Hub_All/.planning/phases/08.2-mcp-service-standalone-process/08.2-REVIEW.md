## Tóm tắt

Phase 8.2 tách MCP server thành tiến trình độc lập (`mcp_service/`) gọi API Service
qua HTTP, đảo decision D-04 của Phase 8.1. Phần lõi bảo mật được xử lý tốt:

- **Forward `X-API-Key` không leak log** — `ApiClient._request` đặt key vào header,
  mọi `logger.error` chỉ ghi `method/path/status_code/error_code`. `auth.py` không
  log giá trị key. Không phát hiện rò rỉ.
- **Validate base URL chống SSRF** — `config.Settings._validate_base_url` ép scheme
  `http`/`https` và bắt buộc có host; có test phủ (`test_config.py`).
- **Map lỗi API sang `ToolError` không leak chi tiết** — `_map_api_error` chỉ dùng
  message từ envelope, không kèm stack trace; có test verify (`test_api_5xx_clean_error`).
- **Hub isolation enforce API-side** — `get_api_key_or_jwt_with_hubs` load `hub_ids`
  từ DB `user_hubs` (không tin payload); `SearchService.intersect_hubs` giao với hub
  user được phép. MCP Service không tự enforce — đúng thiết kế.

Không có Critical. Phát hiện 3 Warning và 5 Info — chủ yếu là edge case xử lý lỗi
và rò rỉ thông tin mức thấp, không phải lỗ hổng chặn release.

## Warnings

### WR-01: `_request` không xử lý 200 + `success=false` và non-200 + `success=true`

**File:** `mcp_service/mcp_app/api_client.py:103-123`
**Issue:** Điều kiện thành công là `resp.status_code == 200 AND envelope.get("success") is True`.
Nếu API Service (hoặc một proxy ở giữa) trả `200` nhưng envelope `success=false` (ví
dụ một handler tương lai trả envelope lỗi với status 200), code rơi vào nhánh lỗi:
`status_code` là 200 nên KHÔNG khớp 401/403/400 → raise `ApiServerError` với message
là lỗi nghiệp vụ thật (ví dụ "Query rỗng" lẽ ra phải là `ApiBadRequestError`).
Việc map lỗi dựa thuần vào HTTP status, bỏ qua `error.code` trong envelope, làm mất
phân loại lỗi khi status và envelope không đồng bộ. Endpoint ask của API Service
hiện trả 400/500 đúng status nên chưa lộ, nhưng đây là giả định ngầm dễ vỡ.
**Fix:** Phân loại lỗi theo cả `error.code` lẫn status, ưu tiên `error.code` khi có:
```python
if envelope.get("success") is True:
    return envelope.get("data")

err = envelope.get("error") or {}
code = err.get("code", "ERROR")
message = err.get("message", "Lỗi không xác định")
logger.error("API Service trả lỗi: method=%s path=%s status_code=%s error_code=%s",
             method, path, resp.status_code, code)

if resp.status_code == 401 or code in ("INVALID_API_KEY",):
    raise ApiUnauthorizedError(message)
if resp.status_code == 403 or code == "FORBIDDEN":
    raise ApiForbiddenError(message)
if resp.status_code == 400 or code == "INVALID_QUERY":
    raise ApiBadRequestError(message)
raise ApiServerError(f"{code}: {message}")
```

### WR-02: Tool `ask_wiki`/`search_wiki` không validate `query` rỗng trước khi gọi HTTP

**File:** `mcp_service/mcp_app/server.py:157-203` (search_wiki), `209-252` (ask_wiki)
**Issue:** `query` được forward thẳng xuống API Service mà không kiểm tra rỗng/whitespace.
Khi LLM client gọi tool với `query=""`, MCP Service vẫn phát một request HTTP, API
Service trả 400 `INVALID_QUERY`, rồi map ngược về `ToolError("INVALID_QUERY: ...")`.
Hành vi cuối cùng đúng, nhưng tốn một round-trip HTTP không cần thiết và phụ thuộc
API Service trả đúng status 400 (xem WR-01). Một guard sớm ở tool cho thông báo rõ
hơn cho LLM client và giảm tải vô ích.
**Fix:** Thêm guard đầu mỗi tool sau `require_api_key`:
```python
if not query or not query.strip():
    raise ToolError("INVALID_QUERY: query không được để trống")
```

### WR-03: `ApiClient` singleton tạo trước khi event loop chạy + cleanup `atexit` mong manh

**File:** `mcp_service/mcp_app/server.py:61-73, 258-278`
**Issue:** `_get_client()` khởi tạo `httpx.AsyncClient` lười ở lần gọi tool đầu tiên —
an toàn. Nhưng `_close_client_atexit` gọi `asyncio.run(_api_client.aclose())` trong
một event loop MỚI, trong khi `httpx.AsyncClient` được tạo và dùng trên event loop
của uvicorn. Đóng một async client gắn connection pool từ một loop khác có thể raise
`RuntimeError: Event loop is closed` hoặc để connection treo. Code đã bọc
`try/except` nên không crash, nhưng connection có thể không được giải phóng sạch khi
process thoát. Vì MCP Service là process sống lâu, đây là rò rỉ chấp nhận được lúc
shutdown, nhưng nên ghi chú rõ là known-limitation hoặc gắn cleanup vào lifespan của
Starlette app thay vì `atexit`.
**Fix:** Ưu tiên gắn shutdown handler vào ASGI app:
```python
def build_asgi_app() -> Any:
    app = mcp.streamable_http_app()
    app.add_event_handler("shutdown", _aclose_client)
    return app
```
với `_aclose_client` là coroutine `await _api_client.aclose()` — chạy đúng trên event
loop uvicorn. Nếu `streamable_http_app()` không cho thêm handler, giữ `atexit` nhưng
ghi rõ trong docstring đây là best-effort known-limitation.

### IN-01: `list_hubs` tool bỏ qua phân trang — chỉ lấy 100 hub đầu

**File:** `mcp_service/mcp_app/server.py:134-136`
**Issue:** Tool gọi `GET /api/hubs` với `params={"per_page": 100}` và không truyền
`page`. Router `list_hubs` cap `per_page ≤ 100`. Nếu một user/API key được assign hơn
100 hub, tool âm thầm chỉ thấy 100 hub đầu (sắp theo `created_at DESC`). Với M2 quy
mô hiện tại khó vượt 100 hub nên đây chỉ là Info, nhưng nên ghi chú giới hạn hoặc
loop phân trang.
**Fix:** Thêm comment nêu rõ giới hạn 100 hub, hoặc lặp `page` đến khi `total` đã lấy hết.

### IN-02: `_to_citations` và mapping search dựa vào `or` cho giá trị số

**File:** `mcp_service/mcp_app/server.py:115` và `198`
**Issue:** `float(c.get("score") or 0.0)` — nếu API trả `score` hợp lệ bằng `0.0`,
`0.0 or 0.0` vẫn ra `0.0` nên không sai kết quả. Nhưng pattern `x or default` cho
giá trị số là dễ gây bug khi `0` là giá trị hợp lệ (ở đây vô hại vì default trùng).
Ghi nhận để nhất quán: dùng `c.get("score", 0.0)` rõ ràng hơn về ý định.
**Fix:** `score=float(c.get("score") if c.get("score") is not None else 0.0)` hoặc
chấp nhận hiện trạng vì default = 0.0 trùng giá trị biên.

### IN-03: `search_wiki` map `document_name` từ `title` — phụ thuộc shape endpoint

**File:** `mcp_service/mcp_app/server.py:194-199`
**Issue:** `/api/search` trả chunk với field `title` còn `/api/ask` citations dùng
`document_name`. Tool xử lý bằng chuỗi fallback `r.get("title") or r.get("document_name")`.
Hoạt động đúng nhưng coupling chặt với shape response của API Service — nếu API đổi
tên field, lỗi chỉ lộ runtime (field rỗng, không exception). Không chặn release;
nên có một integration test ở API-side khoá shape, hoặc một schema chia sẻ.
**Fix:** Ghi chú coupling trong docstring tool, hoặc thêm test contract phía API.

### IN-04: `get_api_key_or_jwt` query `created_by` bằng raw SQL trong khi phần còn lại dùng ORM

**File:** `api/app/auth/dependencies.py:256-263`
**Issue:** Dependency dùng `text("SELECT created_by FROM api_keys WHERE id = :id")`
với named bind param (an toàn injection), nhưng phần còn lại của file dùng SQLAlchemy
ORM `select(User)`. Không phải lỗi — chỉ là không nhất quán pattern. ORM hoá truy vấn
này (qua model `ApiKey`) sẽ đồng nhất hơn và type-safe hơn.
**Fix:** Đổi sang `select(ApiKey.created_by).where(ApiKey.id == principal["id"])` nếu
model `ApiKey` có sẵn; nếu không, giữ nguyên — raw SQL ở đây an toàn.

### IN-05: `mcp_service/tests/__init__.py` rỗng nhưng `api/tests` không có pattern tương ứng

**File:** `mcp_service/tests/__init__.py:1`
**Issue:** File chỉ chứa `from __future__ import annotations` — vô hại, biến `tests/`
thành package. Ghi nhận để nhất quán: kiểm tra xem `pytest` của mcp_service có cần
`tests` là package hay không (rootdir + `testpaths` thường không cần). Nếu không cần,
